# Ralph Meet Desktop App Audit

Date: May 18, 2026  
Scope: Tauri/CEF desktop shell, desktop-only auth/deep-link flow, screen-share capture, stream controls, update pipeline, and desktop dev infrastructure.

## Executive Summary

The desktop app is close to a usable architecture, but a few high-risk areas are still carrying recovery-era complexity:

- Screen sharing has two competing capture models: a custom `xcap` picker feeding Chromium/CEF constraints, and a large Rust native WebRTC bridge that is still exported but no longer the intended path.
- Stream quality/audio/source controls are spread across several UI surfaces, which makes drift likely.
- The updater is configured, but current runtime logs show the release JSON endpoint is not serving valid updater metadata.
- Desktop development now has a cleaner deployed-backend mode, but the config and diagnostics need guardrails so regressions are obvious.
- Desktop auth is functionally improved, but token lifecycle code is duplicated enough that future login flashes or stale-token bugs are likely.

The highest-value work is to make one desktop capture contract authoritative, remove or quarantine the stale native screen-share bridge, and add a small verification harness for picker source IDs, quality updates, and updater metadata.

## Findings

### P0: Custom Picker Source IDs Are Not Proven Chromium/CEF Capture IDs

Evidence:
- `src/hooks/useVoiceChannel.ts:41` maps `window-*` to `window:<id>:0`.
- `src/hooks/useVoiceChannel.ts:46` maps `monitor-*` to `screen:<index>:0`.
- `desktop/src-tauri/src/screen_capture.rs:113` creates monitor IDs from the `xcap::Monitor::all()` array index.
- `desktop/src-tauri/src/screen_capture.rs:157` creates window IDs from `xcap` window IDs.

Impact:
- This is the likely root of "selected application sometimes streams desktop/both monitors." The custom picker IDs are generated by `xcap`, then guessed into Chromium-style `chromeMediaSourceId` strings. That only works if CEF's desktop capture backend uses the same IDs and ordering as `xcap`, which is not guaranteed.
- Monitor selection is especially fragile because `monitor-0` is an enumeration index, while Chromium `screen:<id>:0` expects Chromium's desktop capture source ID, not necessarily our array index.

Recommended fix:
- Make the CEF capture source identifier authoritative. The custom picker can remain, but each selectable source should carry the exact ID accepted by CEF/Chromium capture, not an inferred ID.
- If CEF does not expose those IDs directly through Tauri, add a narrow runtime API in the CEF fork that enumerates desktop capture sources using Chromium/CEF internals and returns `{ id, name, kind, thumbnail }`.
- Add validation after capture: when a selected monitor/window starts, compare `track.getSettings()` and source metadata where possible. If selected-source capture does not match, fail visibly instead of publishing.

Implementation tasks:
- Replace `desktopCaptureSourceId(...)` with an authoritative source ID supplied by the desktop source list.
- Change `ScreenSource` to include `capture_id` separate from display `id`.
- Add telemetry logs for selected source, capture ID, `track.getSettings()`, and final publish track names.
- Add a manual test matrix: single monitor, dual monitor, windowed app, fullscreen app, minimized app, admin/protected window.

### P0: Native Screen-Share Bridge Is Stale But Still Exported

Evidence:
- `desktop/src-tauri/src/lib.rs:9` still includes `mod native_share`.
- `desktop/src-tauri/src/lib.rs:46` still manages `NativeShareState`.
- `desktop/src-tauri/src/lib.rs:175` through `desktop/src-tauri/src/lib.rs:178` still expose native share commands.
- `desktop/src-tauri/src/native_share.rs` still contains a full native capture, encoder, audio, and WebRTC pipeline.

Impact:
- There are two screen-share paths in the codebase: JS/CEF capture and native Rust capture. The active user direction is to use the custom picker with built-in CEF sharing, so the native bridge is now a stale fallback surface.
- Keeping the bridge exported increases maintenance cost, binary complexity, dependency load, and future regression risk. It also makes it easy to accidentally reintroduce a fallback that streams the wrong source.

Recommended fix:
- Decide explicitly whether native sharing is deprecated or experimental.
- If deprecated, remove `native_share.rs`, its commands, and unused dependencies from `desktop/src-tauri/Cargo.toml`.
- If experimental, hide it behind a disabled feature flag and remove all frontend calls from the default app.

Implementation tasks:
- Search for `stop_native_screen_share`, `start_native_screen_share`, `handle_sdp_answer`, and `wait_native_screen_share_connected` after removal.
- Run `cargo check` once the local CEF build issue is resolved.
- Document the single supported desktop capture path in `docs/recovery-refactor-revamp-plan.md` or a dedicated desktop capture ADR.

### P1: Quality/Audio/Source Controls Have Multiple Entry Points

Evidence:
- `src/components/chat/VoiceDashboard.tsx:198` changes stream source.
- `src/components/chat/VoiceDashboard.tsx:205` toggles stream audio.
- `src/components/chat/VoiceDashboard.tsx:224` changes quality.
- `src/components/ContextMenu/LocalMenu.tsx` has stream quality/audio/source actions.
- `src/components/StreamContextMenu.tsx` has another stream control surface.
- `src/components/chat/VoiceChannelView.tsx` bridges picker state into `toggleScreenShare`.

Impact:
- On-the-fly quality and audio changes can drift between the voice dashboard, local context menu, stream context menu, room page, chat page, and DM call paths.
- A UI path may update local state but not the active sender encoding, or preserve `withAudio` while losing source metadata.

Recommended fix:
- Create one `ScreenShareController` hook/API that owns:
  - start selected source
  - stop stream
  - change source
  - change quality
  - toggle stream audio
  - report current status/error
- Make all UI controls call that API rather than shaping options independently.

Implementation tasks:
- Define a shared `ScreenShareOptions` type in one file.
- Replace local inline option types in `VoiceDashboard`, `LocalMenu`, `StreamContextMenu`, `VoiceChannelView`, `RoomPageClient`, `ChatPageClient`, and `DMCallRegion`.
- Add unit tests for option preservation: changing quality must preserve source/audio; toggling audio must preserve source/quality; changing source must preserve quality unless explicitly changed.

### P1: Sender Encoding Updates Need Runtime Verification

Evidence:
- `src/hooks/useVoiceChannel.ts:97` applies track constraints and then calls sender encoding updates.
- `src/lib/sfu-client.ts:949` exposes `updateSenderEncoding(...)`.

Impact:
- Calling `MediaStreamTrack.applyConstraints(...)` changes capture intent, while sender encoding changes RTP output. Both can silently fail or be ignored depending on the browser/CEF backend and negotiation state.
- Current UI can show a selected quality even if sender parameters remain unchanged.

Recommended fix:
- After applying quality, read back `track.getSettings()` and `sender.getParameters()` and log/report whether the requested quality was accepted.
- Surface a non-invasive status if the requested quality is capped.

Implementation tasks:
- Have `updateSenderEncoding(...)` return the applied encoding params or a structured failure.
- Add debug output in one place, not per UI surface.
- Add a "quality effective" field to the local stream state for diagnostics.

### P1: Updater Endpoint Is Configured But Failing At Runtime

Evidence:
- `desktop/src-tauri/tauri.conf.json:70` through `desktop/src-tauri/tauri.conf.json:74` points to `https://github.com/115jon/ralph-meet/releases/latest/download/latest.json`.
- `src/components/UpdateChecker.tsx:30` checks for updates on desktop startup.
- User logs show `Could not fetch a valid release JSON from the remote`.

Impact:
- Desktop users will silently fail update checks every startup.
- This can mask real release problems because the UI intentionally suppresses check failures.

Recommended fix:
- Verify the GitHub release artifact actually publishes `latest.json` with the format expected by Tauri updater v2.
- Add CI validation that downloads the release JSON and verifies signatures/URLs before a release is marked complete.

Implementation tasks:
- Add a small script: fetch updater endpoint, validate JSON fields, check artifact URLs, and fail CI on invalid metadata.
- Consider using a versioned endpoint or stable CDN path instead of only `releases/latest`.
- In dev builds, show updater failures in a diagnostics panel instead of only console warnings.

### P1: Remote Debugging Port Is In Main Desktop Config

Evidence:
- `desktop/src-tauri/tauri.conf.json:25` includes `--remote-debugging-port=9222`.

Impact:
- This is useful during desktop development, but risky if it ships in production builds. It exposes a debugging surface on the local machine and can confuse users with port conflicts.

Recommended fix:
- Move remote debugging into a dev-only Tauri config overlay.
- Keep production config free of debugging flags.

Implementation tasks:
- Create `tauri.dev.conf.json` or extend existing dev command overlay.
- Document how to enable DevTools/remote debugging locally.
- Add a CI check that production config does not contain `remote-debugging-port`.

### P2: Desktop Auth Token Lifecycle Is Duplicated And Noisy

Evidence:
- `src/lib/desktop-auth.ts:10` defines a second desktop token key.
- `src/lib/desktop-auth.ts:53` polls for desktop token handoff.
- `src/lib/desktop-auth.ts:99` syncs Ralph Auth tokens into desktop localStorage.
- Runtime logs show repeated `[DesktopAuth] Token sync tick` and `[DesktopAuth] Persisting desktop token`.

Impact:
- The recent login flash fix is good, but the architecture still depends on localStorage polling and periodic sync.
- Future changes to Ralph Auth storage can desynchronize desktop auth and browser auth.
- Logs are noisy and include token lengths.

Recommended fix:
- Create a dedicated desktop session module with a small state machine: `unknown`, `handoff-waiting`, `authenticated`, `expired`, `signed-out`.
- Centralize token persistence, refresh, and clearing in that module.

Implementation tasks:
- Remove token sync console noise or gate it behind a debug flag.
- Replace polling with event-driven deep-link/session handoff where possible.
- Add tests for startup with existing token, startup without token, deep-link login, expired token, and logout.

### P2: Desktop Vite Proxy Contains Critical Media Behavior Without Tests

Evidence:
- `desktop/vite.config.ts` has custom proxy handling for `/api/attachments` 206 responses.
- It buffers range responses to preserve `Content-Length`.

Impact:
- This is important for Chromium media playback in Tauri, but it is fragile infrastructure embedded in config.
- A Vite/http-proxy upgrade can break attachments or streaming playback.

Recommended fix:
- Extract the proxy response handler into a testable helper.
- Add a regression test for 206 attachment responses with `Content-Range`, `Content-Length`, and no `Transfer-Encoding: chunked`.

Implementation tasks:
- Move proxy response shaping into `desktop/vite-proxy.ts`.
- Add a small Node test using mocked proxy response streams.
- Document why this exists in the desktop dev guide.

### P2: CEF Fork Is A Heavy Operational Dependency

Evidence:
- `desktop/src-tauri/Cargo.toml` depends on Tauri and plugins from `feat/cef`.
- `desktop/src-tauri/tauri-fork` contains local fork changes.
- The desktop app depends on CEF-specific permission and window behavior.

Impact:
- Desktop stability depends on a forked runtime and a local CEF build toolchain.
- Recent `cargo check` attempts were blocked before app code by local CEF dependency/build issues, which means Rust changes are hard to verify quickly.

Recommended fix:
- Treat the CEF fork as first-class infrastructure.
- Add a repeatable setup/check script and fork patch notes.

Implementation tasks:
- Add `docs/desktop-cef-runtime.md` describing fork commits, local setup, build outputs, and known failure modes.
- Add a lightweight `cargo check` path that can validate app code without rebuilding all CEF artifacts, if possible.
- Track fork patches in a patch queue or ADR.

### P2: Desktop Settings Sync Needs Stronger Boundaries

Evidence:
- Rust state lives in `DesktopSettings` in `desktop/src-tauri/src/lib.rs:20`.
- Frontend settings sync through Tauri commands and local stores.
- `start_minimized` is stored but the Rust comment says frontend handles actual minimization.

Impact:
- Settings ownership is split between frontend localStorage and Rust state. This is workable, but not clearly documented.
- Startup behavior can drift if Rust and frontend initialize in different orders.

Recommended fix:
- Define a single desktop settings schema with defaults and ownership notes.
- Add a startup sync acknowledgement from Rust so the frontend can know settings are applied.

Implementation tasks:
- Add a shared TypeScript type plus Rust struct documentation.
- Validate settings payloads at the command boundary.
- Add diagnostics for current desktop settings.

### P3: Screen Source Picker Filtering May Hide Valid Apps

Evidence:
- `desktop/src-tauri/src/screen_capture.rs:16` through `desktop/src-tauri/src/screen_capture.rs:38` blocks many app names.
- `desktop/src-tauri/src/screen_capture.rs:49` filters by minimum visible area.
- Minimized windows are excluded at `desktop/src-tauri/src/screen_capture.rs:134`.

Impact:
- Aggressive filtering improves picker quality, but it can hide apps the user expects to share, especially UWP, protected, minimized, or shell-hosted windows.

Recommended fix:
- Add debug mode in the picker that can show filtered-out windows with reasons.
- Log filter counts and reasons in dev builds.

Implementation tasks:
- Return optional `filterReason` in a debug command.
- Add a "Show hidden/system windows" dev toggle if needed.

## Recommended Remediation Order

1. Stabilize the capture contract: custom picker must return capture IDs that CEF can actually consume.
2. Remove or feature-gate native screen-share bridge code.
3. Centralize screen-share controls and option types.
4. Add quality/audio/source verification logs and tests.
5. Fix updater release metadata and add CI validation.
6. Move remote debugging to dev-only config.
7. Refactor desktop auth into a small state machine.
8. Extract and test desktop Vite proxy media handling.
9. Document the CEF fork and local build process.

## Verification Checklist

- `npm run typecheck`
- `git diff --check`
- Desktop manual test: share monitor 1 only, monitor 2 only, a windowed app, a fullscreen app, and a protected/admin app.
- Desktop manual test: change stream quality while live and confirm `track.getSettings()` plus sender params update.
- Desktop manual test: toggle stream audio while live and confirm audio track state and SFU publish state.
- Desktop updater test: fetch `latest.json`, validate schema, validate artifact URL, validate signature.
- Rust check once CEF local dependency issue is fixed.

## Notes On Current Worktree

This audit was written while the screen-share quality/source fixes were still uncommitted in the parent repo. The findings above treat those changes as current work-in-progress, not as a stable release baseline.
